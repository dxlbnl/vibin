# Proposal 0001 — RFC-2119 specs: normative requirements, scenarios, and a validation gate

- **Status:** Proposed (not yet implemented)
- **Date:** 2026-05-22
- **Affects:** the spec format and the spec → tests → review pipeline
- **Ships as:** migration `0002` (see `migrations/`) once implemented

## Context

Vibin's spec pages (`wiki/specs/<id>-<slug>.md`) currently use **free-form, numbered
acceptance criteria**, and the `## Open questions` section is advisory. Two problems
follow: criteria vary in precision so the `test-writer` has to interpret them, and a spec
with unresolved ambiguity can still flow into implementation.

A comparison against two external spec-driven setups pointed at the fix:

- **Addy Osmani's `agent-skills`** (github.com/addyosmani/agent-skills) — its
  spec-driven-development skill makes specs normative and tests first-class, with explicit
  Boundaries and an Open-Questions discipline.
- **OpenSpec** (github.com/Fission-AI/OpenSpec) — every *requirement* is named and carries
  **≥1 `GIVEN/WHEN/THEN` scenario**, written with **RFC-2119** keywords, and is machine-
  `validate`d (every requirement must have a scenario).

This proposal adopts the *requirement + scenario + RFC-2119 + validation* core, which fits
Vibin's tests-first pipeline almost perfectly: **one scenario maps to one test.** It builds
on the already-shipped Rules-index change (migration `0001`), which put RFC-2119 keywords
into `wiki/architecture.md`; here we extend the same rigor to spec requirements.

## Goal / non-goals

**Goal.** Make spec requirements unambiguous and mechanically testable, and stop vague or
blocked specs from entering the pipeline.

**Non-goals (deferred — see Appendix).** Living/delta specs and an `archive`-merge step; a
spec-validation *hook* (start with a manager-run check); `/status` requirement coverage; an
optional per-spec Design section.

## Design

### 1. Spec format — requirements with IDs, RFC-2119 keywords, and scenarios

Replace the free-form `## Acceptance criteria` in `wiki/specs/README.md` with `##
Requirements`. Each requirement:

```
### B<n>-R<k>: <short name>
The system MUST <single normative statement>.        ← RFC-2119 keyword

- Scenario: <name>
  GIVEN <starting state>
  WHEN <action>
  THEN <observable, testable outcome>
```

- **Stable ID** `B<n>-R<k>` (item id + requirement number), referenced by tests and the
  reviewer.
- **One RFC-2119 keyword** per requirement (`MUST` / `MUST NOT` / `SHOULD` / `MAY`),
  reserved for genuine requirements.
- **≥1 scenario**, each `GIVEN/WHEN/THEN` with an **observable** outcome (no "feels fast").
- Keep `## Context` and `## Out of scope`. Add a one-line RFC-2119 boilerplate note
  (keyword meanings), mirroring `architecture.md`'s Rules section.

### 2. Open Questions — strict and enforced

The `## Open questions` section becomes a gate, not a note:

- The `spec-writer` **MUST** classify every open question as **blocking** or
  **non-blocking**.
- A spec carrying any **blocking** question **MUST NOT** advance to `test-writer`; the
  `spec-writer` **MUST** set `flags: [review]` on the item card and report it, so the
  manager pauses for the user.
- Non-blocking questions are recorded and proceed.

### 3. Manager validation gate

Before dispatching `test-writer`, the manager runs a **spec-validation check** (a manager
step — no new tooling for now; a `PostToolUse` hook can enforce it later):

- every requirement has an ID, exactly one RFC-2119 keyword, and **≥1 scenario**;
- every scenario has an observable `THEN`;
- there are **no unresolved blocking** open questions.

On failure the spec returns to `spec-writer` (it does **not** go forward).

### 4. test-writer / reviewer wiring

- **`test-writer`** writes **one test per scenario**, named by requirement ID (e.g.
  `B3-R1 / happy-path`). A `MUST` scenario is mandatory; a `SHOULD` scenario is tested
  unless explicitly waived in the spec.
- **`reviewer`** verifies **per requirement ID** that each scenario has a passing test, and
  cites IDs in findings (extends today's criterion-by-criterion check).

### 5. Normative convention wording — applied judiciously

State the spec convention normatively where it is a real rule:

- `wiki/INDEX.md` and `.claude/skills/wiki/SKILL.md`: "specs **MUST** use RFC-2119
  requirements with ≥1 scenario each; a blocking open question **MUST** flag the item
  `review`."
- **Do not** MUST-ify ordinary guidance. If everything is `MUST`, the keyword loses its
  weight — reserve the keywords for requirements and rules. (This is the same discipline
  the Rules-index change adopted.)

## Files the implementation will touch

- `wiki/specs/README.md` — the requirement/scenario format (the template the pipeline keys
  on).
- `.claude/agents/spec-writer.md` — emit requirements + scenarios + IDs; enforce the
  Open-Questions classification.
- `.claude/agents/test-writer.md` — one test per scenario, named by ID.
- `.claude/agents/reviewer.md` — verify per requirement ID.
- `.claude/skills/manager/SKILL.md` — the validation gate before `test-writer`; reference
  specific requirement IDs in the `implementer` delegation prompt.
- `wiki/INDEX.md`, `.claude/skills/wiki/SKILL.md` — normative spec convention wording.

## Verification (of the implementation, when it lands)

- A sample feature spec has, for every requirement: an ID, one RFC-2119 keyword, and ≥1
  `GIVEN/WHEN/THEN` scenario with an observable outcome.
- A deliberately vague requirement (no scenario / no keyword) **fails** the manager
  validation and bounces back to `spec-writer`.
- A spec with a **blocking** open question forces `flags: [review]` and the pipeline pauses
  before tests.
- `test-writer` emits one test per scenario named by ID; `reviewer` cites IDs.

## Appendix — related & deferred gains (from the Addy / OpenSpec triage)

Captured so they are not lost; each would be its own later proposal:

- **Living per-domain specs + delta specs (`ADDED`/`MODIFIED`/`REMOVED`) + archive-merge**
  [OpenSpec core] — the real fix for spec staleness; large redesign, likely opt-in per
  project. The biggest open question.
- **Three-tier Boundaries** (`Always` / `Ask-first` / `Never`) seeded into
  `architecture.md`, shipping a default "never commit secrets" [Addy] — cheap safety win.
- **Spec-validation hook** — promote the manager check (§3) into a `PostToolUse` hook.
- **`/status` requirement coverage** — show which requirement IDs have passing tests
  [OpenSpec `view`].
- **Optional `## Design` section** in a spec for architecturally significant items, feeding
  `decisions.md` [Addy PLAN / OpenSpec `design.md`].
- **Item-sizing heuristic** (~5-file cap) in `/intake` / manager [Addy TASKS] — Vibin's
  tests-first already provides per-criterion "verify", so only the sizing heuristic is
  worth borrowing, not a full `tasks.md` layer.

### Already in Vibin (no change needed)

- An `## Open questions` section already exists in the spec format (this proposal makes it
  *strict*, it does not invent it).
- RFC-2119 keywords already live in `architecture.md`'s Rules index (migration `0001`).
- Tests-first is already the pipeline core — stronger than either reference setup.
