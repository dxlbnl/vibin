## Summary

Spec pages used **free-form, numbered acceptance criteria**, and `## Open questions` was
advisory — so criteria varied in precision (the `test-writer` had to interpret them) and a
spec with unresolved ambiguity could still flow into implementation. Specs now state
**requirements**: each carries a stable `B<n>-R<k>` id, **exactly one** RFC-2119 keyword
(`MUST`/`MUST NOT`/`SHOULD`/`MAY`), and **≥1** `GIVEN/WHEN/THEN` scenario with an observable
outcome — **one scenario maps to one test**. `## Open questions` becomes a gate: a
**blocking** question flags the item `review` and stops it advancing. The **manager** runs a
spec-validation check before `test-writer` (no new tooling — a manager step). This extends
the RFC-2119 rigor that migration `0001` put into `architecture.md`'s Rules index to spec
requirements. See `docs/proposals/0001-rfc2119-specs.md`.

## Apply — seed-owned tooling

Adopt the latest seed text for each file below (wholesale if unmodified locally; by hand if
customised). When the upstream seed is available, `git show a1dea5e -- <path>` shows the
exact change.

- `wiki/specs/README.md` — the `## Requirements` format (id + one RFC-2119 keyword + ≥1
  `GIVEN/WHEN/THEN` scenario), the requirement rules, and the Open-Questions gate. This is
  the template the pipeline keys on.
- `.claude/agents/spec-writer.md` — emit requirements + scenarios + ids; classify every
  open question blocking/non-blocking and flag `review` on any blocking one.
- `.claude/agents/test-writer.md` — one test per scenario, named by requirement id; trace
  tests to scenarios.
- `.claude/agents/reviewer.md` — verify **per requirement id**, cite ids in PASS/FAIL
  findings.
- `.claude/agents/implementer.md` — "missing requirement" wording (was "missing criterion").
- `.claude/skills/manager/SKILL.md` — the **spec-validation gate** before `test-writer`
  (id + exactly one keyword + ≥1 scenario + observable `THEN` + no blocking questions; on
  failure the spec returns to `spec-writer`), and the delegation templates that reference
  requirement ids.
- `.claude/skills/tdd-cycle/SKILL.md` — RED step translates each scenario into one test
  named by id; "requirement scenarios become executable".
- `.claude/skills/wiki/SKILL.md` — the normative spec convention (requirements with ids +
  keyword + scenario; blocking question flags `review`).

## Apply — project content (content-aware)

Convert the project's existing spec pages; **never blind-overwrite** a `wiki/specs/*.md`.

1. For each `wiki/specs/<id>-<slug>.md`:
   - Add the one-line RFC-2119 keyword-meaning note under `## Context` (copy from
     `wiki/specs/README.md`).
   - Rename `## Acceptance criteria` to `## Requirements`. Turn each numbered criterion into
     a `### <id>-R<k>: <short name>` block: phrase the statement with **exactly one**
     RFC-2119 keyword, then add **≥1** `GIVEN/WHEN/THEN` scenario whose `THEN` is observable.
     A criterion that already reads as a clear, testable statement usually maps to one
     requirement with one happy-path scenario; split it only if it bundles separate
     obligations.
   - Where a criterion was vague or non-observable, write the scenario you can and record the
     gap under `## Open questions` rather than inventing behaviour.
2. In each spec's `## Open questions`, classify every question **blocking** or
   **non-blocking**. If any is blocking and the item is still open (not in `done/`), add
   `flags: [review]` to its item card so the manager pauses before tests.
3. Do **not** renumber requirements once a spec has shipped (ids are referenced by tests and
   the reviewer). For items already in `wiki/backlog/done/`, converting the historical spec
   is optional — prioritise items still in `ready/`/`doing/`.
